## Objective

Configure the Vite PWA plugin and register a Service Worker so the YGB frontend becomes a fully installable Progressive Web App with offline caching support. This is the foundational infrastructure required by every other Epic 4 frontend story.

Currently `vite-plugin-pwa` is listed in `package.json` but is **not wired into `vite.config.ts`**. Without this, no Service Worker is generated, no caching strategy is applied, and the app cannot work offline.

This issue delivers:
1. PWA plugin configuration in `vite.config.ts` with a `manifest.json` (name, icons, theme colour, display mode `standalone`).
2. A Service Worker generated via `vite-plugin-pwa`'s `generateSW` strategy (Workbox), caching:
   - App shell (HTML, JS, CSS bundles) — `CacheFirst`.
   - The Uganda location dataset JSON (`/api/v1/locations/dataset`) — `StaleWhileRevalidate`.
3. Service Worker registration in `src/main.tsx` using the `registerSW` utility exported by `vite-plugin-pwa`.
4. Offline fallback: when the app shell cannot be fetched, the cached version is served transparently.

## Architectural Context

- **Frontend (Infrastructure / Configuration)**:
  - `vite.config.ts` — add `VitePWA({ ... })` plugin.
  - `src/main.tsx` — import and call `registerSW()` from `virtual:pwa-register`.
  - `public/manifest.webmanifest` (auto-generated by the plugin from config).
  - No changes to application-layer or domain-layer files.

## Technical Constraints & Clean Code

- **Plugin version**: Use the `vite-plugin-pwa` version already declared in `package.json` (`^1.3.0`). Do not upgrade without confirming the existing Vite 8 compatibility.
- **generateSW strategy**: Use `generateSW` (not `injectManifest`) to keep the configuration declarative and minimal.
- **File limits**: `vite.config.ts` must remain readable and under 60 lines. Extract complex Workbox runtime caching rules into a `workbox-config.ts` helper if needed.
- **No framework logic in config**: The PWA config is pure build-time configuration — no React/Zustand imports.
